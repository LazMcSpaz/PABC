# v0.2+ Implementation Roadmap

Sequencing plan for the v0.2+ design work into phases a coding agent can
own one at a time. **This is a map, not a recipe** — it tells you the
order, the dependencies, the files each phase touches, and the
cross-system gotchas to watch. The **rules** live in the spec; read the
cited sections before planning your phase.

- **Design of record:** `docs/mechanical-spec-v0.1.md` (§16–§20).
- **Existing detailed plans:** `docs/demo-gameplay-v0.2-plan.md` (§16,
  already implemented) and `docs/tech-wheel-plan.md` (§17, your Phase 1
  instructions).
- **Editor contract:** `docs/content-schema-v0.1.md`.

## Guardrails (every phase)

- Work on branch **`claude/ashland-conquest-demo-BViiz`**. No PRs unless
  asked.
- **Never edit `content/` or `src/game/content/`** — the content authoring
  tool owns those and regenerates them. `src/game/content.js` and
  `config.js` are engine registries you *may* extend (add schema fields,
  stub chips, constants), but treat authored **values** as the content
  team's; coordinate, and keep new tunables in `config.js`.
- **Keep the demo playable after every phase.** No broken intermediate
  states — each phase ships a working game.
- **Verify before pushing:** run the harness (`src/game/harness.js`) and
  drive the prototype in a browser. Add harness coverage for your phase.
- **TBD numbers:** the spec leaves constants as TBD on purpose. Pick
  sensible values, put them in `config.js`, leave them tunable, and note
  your choices in the commit.
- The **engine↔UI seam is `src/prototype/engineAdapter.js`** — expose new
  state there rather than reaching into engine internals from React.

## Why this order (dependency logic)

The single most important call: **§18 is not one block.** It contains three
things of wildly different size and dependency, so it is split across three
phases:

- **Loyalty (§18.2)** — small, foundational. **§20 and §19 both depend on
  it**, so it goes early.
- **Influence / ZoC (§18.3)** — medium; needs Loyalty; **§19's vision and
  §18's diplomacy reception depend on it**.
- **Diplomacy (§18.4–18.13)** — huge AI-heavy capstone; reads *everything*
  (Loyalty, Influence, fog for intel-trade, economy for scrap-deals).

Other key edges:
- **§17 `techLevel`** is read by Market gating now and by **§20 build-gating**
  later — but **§20 retires the Market**, so do *not* over-invest in §17's
  market-tier plumbing; it's interim.
- **§20** needs only Tech Level + Loyalty, so it slots in right after them —
  and retiring the Market early reduces churn for later phases.
- **Fog (§19)** needs Influence/ZoC (vision) and §17 (Intelligence path).
- **Diplomacy (§18C)** comes last because intel-deals need fog and
  scrap-deals need the economy.

```
§16 ✓ ── ambush term ──────────────┐
§17 Tech ──┬─ Intelligence path ───┐│
           └─ techLevel gate ──┐   ││
§18.2 Loyalty ──┬──────────────┤   ││
                │              ▼   ▼▼
                ├─► §20 Economy (Tech×Loyalty gate, retires Market)
                └─► §18.3 Influence/ZoC ─► §19 Fog ─► §18.4-13 Diplomacy
```

Phase order: **0 → 1 (§17) → 2 (§18.2) → 3 (§20) → 4 (§18.3) → 5 (§19) →
6 (§18.4–13).**

## Phase 0 — Groundwork & regression safety net

**Goal:** lock a baseline so later phases can't silently break the demo.

- Run `harness.js`; confirm §16 behaviors (attrition, salvage, veterancy,
  movement budget) and a full AI game complete cleanly. Capture a baseline
  log.
- Confirm the verify loop: harness for engine logic + headless browser for
  UI (the prototype is Vite/React; a local Chromium is available for
  Playwright when network installs are blocked).
- Catalogue the **radial visual language** to reuse — the `ControlMeter`
  and `ContestOverlay` drawing styles are the template for the tech wheel
  (Phase 1), the 8-slice Loyalty pie (Phase 2), and influence overlays
  (Phase 4). Read these before building any new radial widget.
- Decide the home for TBD constants: **`config.js`**.

**Done when:** a documented baseline exists and the verify loop is proven.

## Phase 1 — Tech Wheel (§17)

**Primary instructions already exist: `docs/tech-wheel-plan.md`.** Follow
it; the spec **§17** is the rationale behind it.

- **Depends on:** nothing (foundational).
- **Scope:** replace `player.tech` with `research` / `permanentResearch` /
  `techLevel` / `techWheel`; new **`src/game/tech.js`** node registry;
  `recomputeTech` → `recomputeResearch`; `CONFIG.tech` thresholds
  (`researchThresholds [2,4,6,8]`, `marketTierByLevel`); the four **entry
  effects** at their use sites; assign/unassign (no Action cost) + the LIFO
  peel on level drop.
- **Touches (engine):** `stats.js`, `setup.js`, `config.js`, `actions.js`
  (`unlockedTier`), `turn.js`, `contest.js` (Military +1), `encounters.js`
  (Intelligence redraw, stacks with `recon-team`), `events.js`
  (`research_changed`, `tech_level_changed`, `tech_node_assigned`,
  `tech_node_lost`), `content.js` (`labs` → `research:1`; new
  `advanced-lab` `research:2`).
- **Touches (UI):** new `TechWheel.jsx` (radial); `engineAdapter.js` exposes
  `research` / `techLevel` / `abilityPointsAvailable` / `techWheel`.
- **Interactions & gotchas:**
  - The **Economy entry effect** (+scrap per held location) lives in
    `turn.js collectProduction` — **Phase 3 replaces `collectProduction`
    with the Output slider**, so that effect must be re-homed then. Leave it
    cleanly isolated.
  - **Market gating here is interim** (Phase 3 reframes it as build-gating).
    Keep the `{techLevel ≥ 1/3/5}` mapping factored out so Phase 3 can reuse
    it.
  - `town-hall` ("+1 foothold cap") becomes meaningless after Phase 2 (fixed
    Loyalty ceiling 8) — leave a note; it'll be repurposed.
- **AI:** `ai.js` should spend Ability Points (assign wheel nodes) and value
  Research sources.
- **Done when:** harness in §17's "Verify" passes; the wheel is assignable
  in-browser and entry effects fire.

## Phase 2 — Loyalty (§18.2, with §18.0 terminology)

**Goal:** replace foothold/decay with the 8-slice **Loyalty** pie. Small but
foundational — §20 and §19 read it.

- **Depends on:** none mechanically (do after §17 so the radial UI pattern
  is established).
- **Spec:** §18.0 (canonical terms), §18.2; supersedes §6.3.2.
- **Scope:** replace the signed foothold `F` with `loyalty` (0–8). **Control
  flips only** (a) on a lost contest (§6.3.1, unchanged) or (b) when
  `loyalty == 0` → peel one Control section to neutral per Upkeep until
  fully neutral. Loyalty rises while held/integrated, decays when neglected,
  floors at 0, ceiling fixed at 8. Capital stays inert. Emit a
  **danger-warning before any peel**.
- **Touches (engine):** `contest.js` (capture + peel logic), `turn.js`
  (Upkeep Loyalty tick — replaces the foothold tick), `setup.js` (init),
  `config.js` (rise/decay rates, danger threshold, peel cadence), `stats.js`
  if Loyalty feeds derived values, `events.js` (`loyalty_changed`,
  `loyalty_failing`, `control_peeled`).
- **Touches (UI):** extend `ControlMeter` with the 8-slice center pie; a
  "loyalty failing" alert (`EventFeed.jsx` + a meter cue); `engineAdapter.js`
  exposes `loyalty`.
- **Interactions & gotchas:**
  - This is the seam Phase 3 (gating), Phase 4 (influence), and Phase 6
    (development/diplomacy) all read — keep `loyalty` clean and queryable.
  - `town-hall` chip is now a Loyalty-**rate** chip conceptually
    (content-owned; coordinate, don't author values).
- **AI:** `ai.js` should value keeping units on freshly captured Locations to
  build Loyalty.
- **Done when:** harness shows a neglected capture peeling back to neutral
  via Loyalty 0, a garrisoned one holding, and the warning firing first.

## Phase 3 — Economy & City Development (§20) — retires the Market

**Goal:** chips become the output of the economy. Most disruptive phase
(changes how players get chips) — budget accordingly and keep the demo
playable.

- **Depends on:** Phase 1 (Tech Level gate) + Phase 2 (Loyalty ladder).
- **Spec:** §20 (all); read §20.6 (gating display *contract*) and §20.11
  (engine map) closely. Forward-pointers sit at §4.1 and §17.2.
- **Scope:** remove the Market + Acquire; add per-Location **Output** + the
  **build slider** (bank vs. construction); build chips into slots and
  **upgrade in place**; unit-chips require a stationed unit; **rush-building**
  with banked scrap; **Tech Level × Loyalty** gating with the §20.6 display
  rule; **selective per-chip `upkeep`** (dormant via the `disabled` flag if
  unpaid, not destroyed).
- **Touches (engine):** **remove** `market.js` / `churnMarket` (`turn.js`),
  the Acquire handler (`actions.js`), market seeding (`setup.js`), and the
  `marketRow`/`marketDeck` zones (`effects.js`). **Add** Output + slider +
  `activeBuild` to Location state; rework `turn.js` Upkeep order (§20.11) so
  the slider **replaces `collectProduction`** — **and re-home the §17 Economy
  entry effect here**. New effects (`START_BUILD`, `UPGRADE_CHIP`,
  `RUSH_BUILD`, `SET_BUILD_SLIDER`) + events. `content.js` chip schema gains
  `buildCost` / `upgradesTo` / `loyaltyReq` / `upkeep` (add fields + stub
  chains like `labs`→`advanced-lab`; values are content-owned). Reuse the
  Phase 1 `{techLevel ≥ 1/3/5}` map for build-gating.
- **Touches (UI):** big — slot-click **build menu** + chip-click **upgrade**
  view per §20.6, the slider control, dormant-chip indicator; **remove all
  Market UI**; `engineAdapter.js` exposes buildable/upgradeable sets (already
  filtered by Tech Level, with Loyalty-locked greyed + reason).
- **Interactions & gotchas:**
  - `collectProduction` is gone — make sure every old scrap source (the §17
    Economy node, scrap-production chips) now flows through Output.
  - Capture/loss + Loyalty-drop rules (§20.8) interlock with §6.3.3 and the
    §17.3 peel — keep "newest = last built/upgraded".
  - The AI **must** be able to set sliders and pick builds or the demo
    stalls.
- **AI:** meaningful lift in `ai.js` (slider policy, build priorities, rush
  decisions).
- **Done when:** a full game runs with no Market — players build/upgrade
  chips off Output, rush works, an unpaid upkeep chip goes dormant, and the
  Tech×Loyalty display rule behaves in-browser.

## Phase 4 — Influence & Zone of Control (§18.3)

**Goal:** the soft territorial field. Light-touch by decision — do **not**
wire it into contest math or yield.

- **Depends on:** Phase 2 (Loyalty feeds local influence).
- **Spec:** §18.3 (+ §18.0); note the explicit **ZoC ≠ Vision** warning —
  keep these as separate sets (Phase 5 builds Vision).
- **Scope:** a deterministic per-faction **Influence** scalar field + a
  derived **ZoC owner map**, recomputed (no dice) on control/Loyalty/chip
  change. Light-touch uses: reinforcement routing (connect to the existing
  §16.5 friendly/neutral pathing), encounter-reveal `condition` hook;
  diplomatic reading is **deferred to Phase 6**.
- **Touches (engine):** new recompute pass (sibling to `recomputeStats` /
  `recomputeResearch`, e.g. `influence.js`) writing into `state.world`;
  `config.js` (range, falloff, dominance threshold); `events.js`
  (`zone_changed`); `setup.js`. Influence chips are content.
- **Touches (UI):** a ZoC overlay on the map (reuse the radial/territory
  drawing language); `engineAdapter.js` exposes the per-faction ZoC map.
- **Interactions & gotchas:** keep the field cheap to recompute and the ZoC
  set distinct from any future Vision set; resist the deferred uses
  (contest/yield) — they were intentionally left out.
- **AI:** minor (reads ZoC for routing).
- **Done when:** capturing/integrating Locations visibly shifts ZoC borders,
  and reinforcement routing respects them.

## Phase 5 — Exploration & Fog of War (§19)

**Goal:** the fourth X. Compute-heavy and the most invasive to the AI —
**AI plays under the same fog.**

- **Depends on:** Phase 4 (ZoC projects vision), Phase 1 (Intelligence
  path), Phase 2 (Loyalty scales Location vision). Ambush touches the
  already-shipped §16 combat — don't regress it.
- **Spec:** §19 (all); §19.11 is the engine map; §19.2 governs what
  persists vs. goes stale.
- **Scope:** per-faction **`state.visibility[fid]`** (`explored` / `visible`
  / `memory` + **ghosts**); **Vision** as a unit stat + Location/ZoC
  contribution; **Detection** as a separate short range; LoS over
  elevation/cover; **concealment + hidden armies**; **ambush** (adds a term
  to the §16.6 total and suppresses the §10 reaction window for the
  surprised side); hidden encounter hexes; scout-loadout + Watchtower chips
  (content); the Intelligence-path branch effects (vision / espionage).
- **Touches (engine):** new `visibility.js` (LoS recompute, **incremental on
  move** — the scale guard); `board.js` (elevation/cover flags — coordinate,
  terrain data may be content); `contest.js` (ambush term + reaction-window
  suppression); `actions.js` (Move reveals; concealment check at contest
  declaration); `events.js` (`hex_explored`, `unit_spotted`,
  `unit_lost_sight`, `ambush_triggered`); new effects (`REVEAL_REGION`,
  `GRANT_VISION`, `PLANT_FALSE_INTEL`).
- **Touches (UI):** big — three-state fog rendering, dimmed **ghosts**, the
  concealment/detection cues; `engineAdapter.js` serves the *viewing
  player's* visibility only.
- **Interactions & gotchas:**
  - **AI fog refactor is the heavy part** — `ai.js` must plan on its own
    `visibility[fid]` + memory and tolerate acting on stale ghosts. No
    vision cheats at baseline (§19.10).
  - Build for the **larger map** from the start (incremental recompute);
    don't key anything off the 30-hex test field.
  - Keep ZoC (Phase 4) and Vision distinct; Vision can exceed ZoC, and a
    concealed enemy can sit inside your ZoC unseen.
- **Done when:** fog renders per faction, ghosts persist, an ambush from fog
  resolves with the bonus + no defender reaction, and an AI demonstrably
  scouts and gets surprised.

## Phase 6 — Faction model, Diplomacy, Coalitions, Vassalage & Recognition (§18.4–18.13)

**Goal:** the capstone. Largest AI lift in the project; reads every prior
phase.

- **Depends on:** Loyalty (§18.2), Influence (§18.3, reception), Fog (§19,
  intel-deals + shared vision), Economy (§20, scrap-deals), Tech (§17).
- **Spec:** §18.4–§18.13. §18.5 (state), §18.6 (deal primitive), §18.7
  (verbs), §18.8 (AI + coalitions), §18.9 (vassalage), §18.10 (Recognition
  win), §18.12 (engine map).
- **Scope:** non-cosmetic **faction model** (temperament, trustworthiness,
  grudge, sociability, victory lean, expansion appetite, coveted targets) +
  **faction↔faction Standing** (extend `state.factionStanding`); player
  **Menace** + **Honor**; derived Tolerance/trust-floor; the **deal basket**
  primitive; the action verbs; **AI valuation/acceptance + AI-to-AI politics
  + coalitions** (threat = both Menace *and* power/VP lead); **vassalage**
  (tribute, resentment, rebellion); the **Recognition** victory track.
- **Touches (engine):** **extend `standing.js`** (it already holds §15
  player↔faction standing) to faction↔faction; new `diplomacy.js` (deals,
  pacts, coalitions, vassals, `state.diplomacy`); win-condition evaluator
  (Recognition gate); **reuse §15 encounter delivery** (`encounters.js`,
  `private` mode + `ctx.interact`) for incoming proposals/pact-calls; reuse
  `ADJUST_STANDING`; many new effects/events (§18.12). Faction characteristic
  tables are content.
- **Touches (UI):** a new **Diplomacy screen** (outgoing proposals; the
  incoming ones arrive as encounters); reputation/recognition readouts;
  `engineAdapter.js` exposes standing/menace/honor/recognition.
- **Interactions & gotchas:**
  - The `intel` deal items need **Phase 5** (vision/mapData/shared vision);
    influence-reception needs **Phase 4**; scrap-deals/gifts/tribute need
    **Phase 3** (scrap economy).
  - `ai.js` grows the most here — one valuation engine drives AI answers,
    AI offers, AI-to-AI diplomacy, and coalition formation. Budget for it.
  - Coalitions are the anti-runaway brake; ensure they read both Menace and
    power lead and that they apply to the diplomacy player too (§18.8/18.10).
- **Done when:** an AI-vs-AI war forms and resolves without the human;
  Menace/Honor move on aggression/broken promises; a coalition forms against
  a runaway; a vassal can be taken and can rebel; and the Recognition victory
  can be reached and is correctly gated by Menace/Honor.

## Cross-cutting (applies to every phase)

- **AI (`ai.js`)** grows each phase; Phases 3, 5, and 6 are the big lifts.
  Never let a phase leave the AI unable to play the new system, or the demo
  stalls.
- **Adapter (`engineAdapter.js`)** is the only place React reads engine
  state — add new exposures there, and remember Phase 5 makes the adapter
  **per-viewing-player** (fog).
- **Events:** every phase adds names to `events.js`; surface the meaningful
  ones in `EventFeed.jsx`.
- **Radial visual language:** tech wheel, Loyalty pie, influence overlay, and
  the contest overlay should share the `ControlMeter`/`ContestOverlay`
  drawing style — it's the game's signature UI identity.
- **The big 4X UI re-layout** (resource ribbon, docked context panel, lens
  screens for the wheel/diplomacy) is a **separate UI track** that can run in
  parallel but is best planned once Phases 3–6 produce the data it surfaces.
- **Testing:** extend `harness.js` per phase for engine logic; drive the
  prototype in a browser for feel/regressions. A type-check/test pass proves
  *correctness*, not *feature* — confirm the feature in the app.
- **Content boundary (repeat):** `content/` and `src/game/content/` are
  off-limits; add engine schema fields/stubs in `content.js`/`config.js`
  only, and coordinate on authored values.
